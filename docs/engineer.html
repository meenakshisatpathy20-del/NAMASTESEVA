PROJECT OVERVIEW

What is NamasteSeva modelling?

NamasteSeva models the internal domain architecture of a restaurant platform where one application coordinates three independent business workflows while keeping object creation, business entities, lifecycle management, interchangeable behavior, and notification responsibilities separated.

SYSTEM OBJECTIVE

Translate real restaurant workflows into maintainable Java objects.

A real restaurant application contains multiple entities and workflows. Customers select restaurants, browse menus, create carts, make payments, place orders, reserve tables, configure catering events, and receive notifications.

Instead of placing these responsibilities inside one large class, NamasteSeva divides the system into domain models, factories, managers, strategies, lifecycle enums, and a separate notification subsystem.

01
Customer selects a service.

The application supports three separate workflows: food ordering, table reservation, and party/catering booking.

02
Domain objects capture the request.

Objects such as Cart, Order, TableBooking, EventDetails, CateringPackage, and PartyBooking represent real business data.

03
Payment behavior executes where required.

Food ordering uses the PaymentStrategy abstraction to execute Card, UPI, or Net Banking payment behavior.

04
Factories create complex objects.

FoodOrderFactory creates Order objects while PartyBookingFactory creates PartyBooking objects from the references passed to them.

05
Managers store and coordinate objects.

OrderManager manages Order objects. BookingManager manages table and party bookings. RestaurantManager maintains restaurants.

06
Lifecycle values describe current state.

OrderStatus and BookingStatus enums restrict state to known values and allow managers to coordinate lifecycle changes.

07
Business events can create notifications.

Order and booking events can be converted into notification objects, decorated with additional information, and published to the notification system.

08
Notification strategies deliver the message.

The NotificationEngine delegates delivery behavior to EmailStrategy, SMSStrategy, or PopupStrategy without changing notification construction.

DESIGN PATTERN MAP

Patterns used in the current LLD.

NamasteSeva applies design patterns where responsibilities need to be separated, behavior must remain interchangeable, or infrastructure should be reusable across workflows.

CREATIONAL

Factory Pattern

Separates creation of important business objects from the application flow.

Main → Factory → Business Object
OrderFactory FoodOrderFactory PartyBookingFactory
CREATIONAL

Singleton Pattern

Ensures that selected centralized components expose one shared instance to the application.

Application → getInstance() → Shared Object
Logger Manager classes where implemented as Singleton
BEHAVIORAL

Strategy Pattern

Allows payment and notification delivery behavior to change at runtime through common interfaces.

Interface → Runtime Strategy → Execution
PaymentStrategy CardPayment UPIPayment NetBankingPayment INotificationStrategy EmailStrategy SMSStrategy PopupStrategy
STRUCTURAL

Decorator Pattern

Adds timestamp and signature information to notification objects without modifying SimpleNotification.

Simple → Timestamp → Signature
INotification SimpleNotification NotificationDecorator TimestampDecorator SignatureDecorator
BEHAVIORAL

Observer Pattern

Allows registered observers to react when notification state changes inside the observable component.

Observable → notifyObservers() → Observers
IObservable IObserver NotificationObservable NotificationEngine
